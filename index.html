Bases de datos relacionales

SQL

El lenguaje universal para consultar, organizar y extraer información estructurada.

SQL — Structured Query Language. Lenguaje para consultar bases de datos relacionales — se refiere a información estructurada presentada mediante tablas con filas y columnas, las cuales se conectan entre ellas por primary keys.
¿Qué es una base de datos relacional?
Al crear una base de datos, se usan tablas separadas para diferentes tipos de entidades, lo que lleva a la necesidad de relacionar estas tablas entre sí.

Por ejemplo: una tabla de Clientes, una de Pedidos y una de Productos — cada una guarda su propia información, pero se conectan a través de llaves (keys) para poder cruzar los datos cuando se necesita.

Los sistemas de administración de bases de datos relacionales usan SQL precisamente para almacenar la información en estas tablas separadas y, al mismo tiempo, tener un lenguaje común para manipular estas tablas al hacer un reporte o análisis.
Tipos de relaciones entre tablas
Uno a uno (1:1)
Cada registro en la tabla A se relaciona con exactamente un registro en la tabla B.
Ej: Un usuario tiene un único perfil.
Uno a muchos (1:N)
Un registro en A puede relacionarse con muchos en B, pero cada registro en B pertenece a uno solo en A.
Ej: Un cliente tiene muchos pedidos.
Muchos a muchos (N:M)
Muchos en A se relacionan con muchos en B. Se resuelve con una tabla intermedia que guarda los pares de IDs, convirtiendo la relación N:M en dos relaciones 1:N.
Ej: Un estudiante toma muchos cursos; un curso tiene muchos estudiantes.
¿Cómo funciona la tabla intermedia? En lugar de intentar guardar muchos cursos dentro de la fila de un estudiante (lo cual no es posible en SQL), se crea una tabla estudiante_curso con dos columnas: estudiante_id y curso_id. Cada fila representa una inscripción — por ejemplo, la fila (3, 7) significa que el estudiante 3 está inscrito en el curso 7. Así, buscar todos los cursos de un estudiante es simplemente filtrar esa tabla por su ID.
Auto-referente
Una tabla se relaciona consigo misma — un registro apunta a otro registro de la misma tabla.
Ej: Un empleado tiene un manager que también es empleado.

Comandos esenciales básicos
Tabla de ejemplo — la usamos en todos los comandos de abajo

SELECT · FROM · WHERE Elegir y filtrar datos
Le dice a la base de datos qué columnas mostrar (SELECT), de qué tabla (FROM) y bajo qué condición (WHERE).
SELECT nombre, pais FROM clientes WHERE pais = 'MX';
Resultado sobre la tabla de ejemplo: de las 5 filas, solo se devuelven las de Ana, Luis y Sofía (las que tienen pais = 'MX'), y de cada una solo se muestran las columnas nombre y pais — no ventas, porque no se pidió.
GROUP BY + COUNT/SUM/AVG Agregar datos
Junta todas las filas que comparten un mismo valor en una columna (por ejemplo, todas las filas con pais = 'MX') y las convierte en una sola fila de salida por grupo, con un cálculo hecho sobre ese grupo — contar cuántas hay, sumar un total, o sacar un promedio.
SELECT pais, COUNT(*) AS total_clientes FROM clientes GROUP BY pais;
Resultado sobre la tabla de ejemplo: en vez de las 5 filas originales, obtienes solo 2 filas — una por país: "MX, 3" (Ana, Luis y Sofía) y "CO, 2" (Carla y Mateo). COUNT(*) cuenta cuántas filas cayeron en cada grupo; si en vez de COUNT usaras SUM(ventas), te daría el total de ventas de cada país en lugar del número de clientes.
ORDER BY · LIMIT Ordenar y recortar
Ordena el resultado (ascendente o descendente) y, si solo quieres los primeros N, lo recortas con LIMIT.
SELECT nombre, ventas FROM clientes ORDER BY ventas DESC LIMIT 5;
Resultado sobre la tabla de ejemplo: ORDER BY ventas DESC acomoda a los 5 clientes de mayor a menor venta: Sofía (2100), Ana (1200), Luis (800), Carla (450), Mateo (300). Si en vez de LIMIT 5 pusieras LIMIT 2, solo verías a Sofía y Ana — el "top 2" de ventas.
HAVING Filtrar grupos
Es como WHERE, pero se aplica después de agrupar — filtra grupos completos, no filas individuales. WHERE no puede usarse aquí porque WHERE filtra antes de que existan los grupos.
SELECT pais, COUNT(*) AS total FROM clientes GROUP BY pais HAVING COUNT(*) > 100;
Resultado sobre la tabla de ejemplo (con un umbral más bajo, > 2, para que se note): primero se agrupa por país igual que con GROUP BY ("MX, 3" y "CO, 2"), y luego HAVING revisa cada grupo ya formado — solo se queda con "MX, 3" porque 3 > 2; "CO, 2" se descarta porque no supera el umbral.
JOIN Unir tablas — el más usado de todos
Cuando la información que se necesita está repartida en dos tablas distintas (por ejemplo, Clientes y Pedidos), JOIN las une usando la llave que comparten. Es probablemente el comando más usado en el día a día de un analista, porque casi ningún reporte real vive en una sola tabla.
Ejemplo práctico: imagina una segunda tabla, pedidos, con columnas id, cliente_id y monto: la fila (id=1, cliente_id=1, monto=300) significa "el pedido 1 lo hizo el cliente con id 1" (Ana, según la tabla clientes de arriba).
SELECT clientes.nombre, pedidos.monto FROM clientes JOIN pedidos ON clientes.id = pedidos.cliente_id;
Resultado: SQL busca, para cada fila de pedidos, el cliente cuyo id coincide con cliente_id, y arma una nueva fila combinada. El pedido con cliente_id=1 y monto=300 se convierte en la fila "Ana, 300" — aunque "Ana" vivía en una tabla distinta a "300".

¿Para qué sirve SQL?
Extraer, filtrar, agrupar y relacionar información estructurada con precisión. Es la herramienta central de cualquier analista de datos.
Reportes de campaña Análisis de ventas Segmentación de usuarios KPIs de producto Auditorías financieras
¿Para qué NO sirve?
Para datos sin estructura fija, escala masiva en tiempo real, o archivos multimedia.
Ventajas
Lenguaje estándar universal — aprendido una vez, funciona en casi todo
Relaciones complejas entre tablas con JOINs precisos
Compatible con BigQuery, Looker, Power BI, Python
Desventajas
Esquema rígido — difícil de cambiar una vez definido
JOINs masivos entre tablas grandes pueden ser lentos
Escala vertical (más detalle en SQL vs NoSQL)
No ideal para datos no estructurados o variables
Bases de datos no relacionales

NoSQL

Bases de datos diseñadas para flexibilidad, velocidad y escala masiva. No reemplazan a SQL — resuelven problemas distintos.

NoSQL — "Not Only SQL". Familia de bases de datos que no usan tablas fijas. Los datos se almacenan en formatos flexibles: documentos JSON, grafos, pares clave-valor o columnas, pensados para datos no estructurados.
¿Para qué sirve NoSQL?
Para casos donde la flexibilidad, la velocidad y la escala importan más que la consistencia estricta. Ideal cuando los datos cambian de forma frecuentemente o cuando millones de usuarios acceden al sistema simultáneamente. Ejemplo real: en una app como WhatsApp o un sensor de temperatura, llegan miles de registros nuevos por segundo. Esperar a que una tabla SQL verifique relaciones y reglas estrictas en cada uno sería demasiado lento — NoSQL los guarda "tal cual llegan" y resuelve el orden después.
Apps en tiempo real Redes sociales IoT y sensores* Cachés de sesión* Recomendaciones
*IoT y sensores: "Internet de las Cosas" — dispositivos físicos (termostatos, cámaras, relojes inteligentes) que envían datos a internet constantemente, sin intervención humana.
*Caché de sesión: guardar temporalmente datos de "quién está conectado y qué está haciendo ahora" (ej. tu carrito de compras), para no recalcularlos cada vez que cargas una página.
¿Para qué NO sirve?
Cuando necesitas relaciones complejas entre entidades, o consistencia absoluta (ej. transacciones bancarias). Por qué: un banco necesita que el dinero que sale de una cuenta y entra a otra ocurra como un solo paso indivisible — si algo falla a la mitad, todo debe revertirse. SQL está construido para garantizar justo eso; NoSQL prioriza velocidad sobre esa garantía estricta.
Los 4 tipos de NoSQL

Cada tipo organiza la información de forma distinta. Así se ve en la práctica:

Documentos
Como una ficha por persona: cada documento JSON* guarda lo que necesite, sin que todos tengan los mismos campos. El perfil de Ana puede tener "plan" y el de Luis no.
Ejemplo real: Instagram guarda tu perfil así — nombre, fotos, bio — cada perfil es flexible y no depende de una tabla rígida.
Clave-valor
Como un guardarropa con tickets: le das un número (clave) y te devuelven tu abrigo (valor) al instante. No busca nada, solo "entrega lo que está en esa casilla".
Ejemplo real: cuando inicias sesión en una app, el servidor guarda "token123 → tus datos de sesión" para no pedirte la contraseña en cada clic.
Columnar
En vez de leer fila por fila, lee columna completa de golpe. Si solo quieres "todos los clicks", no toca nada más — mucho más rápido para sumar millones de datos.
Ejemplo real: YouTube necesita contar cuántas vistas tuvo un video entre miles de millones de registros — leer solo la columna "vistas" es muchísimo más rápido que leer fila por fila.
Grafos
Piensa en un mapa de conexiones, no en una tabla: cada persona o cosa es un punto, y las líneas son sus relaciones. Seguir esas líneas es instantáneo.
Ejemplo real: cuando Facebook sugiere "personas que quizá conozcas", recorre las conexiones de tus amigos — eso es un grafo en acción.
*Documento JSON: un bloque de texto con forma de "etiquetas y valores" — ej. {"nombre":"Ana","edad":28} — fácil de leer tanto para humanos como para programas.
Ventajas
Flexible — no requiere esquema definido de antemano
Muy rápido para patrones de acceso simples
Ideal para datos en tiempo real y alta concurrencia
Desventajas
Sin estándar universal — cada BD tiene su propio lenguaje
JOINs son difíciles o inexistentes
Consistencia eventual — datos pueden estar desactualizados por milisegundos
Duplicación de datos — menos eficiente en almacenamiento

Dos palabras técnicas que vale la pena entender

La regla de decisión
Usa SQL cuando: los datos tienen estructura predecible, necesitas relaciones entre entidades, o requieres reportes complejos y auditorías.

Usa NoSQL cuando: los datos son flexibles o variables, necesitas velocidad extrema en tiempo real, o tu sistema debe escalar a millones de usuarios simultáneos.

Usa ambos (arquitectura híbrida): Google, Netflix y Spotify usan SQL para analytics y reporting, y NoSQL para el producto en tiempo real. No son mutuamente excluyentes.
Archivos de datos

CSV y formatos de intercambio

El CSV es el formato universal de datos — simple, portable y compatible con todo. Pero tiene limitaciones. Conocer cuándo usarlo y cuándo no es una habilidad fundamental.

CSV (Comma-Separated Values): archivo de texto plano donde cada fila es un registro y las columnas están separadas por comas. Es el "mínimo común denominador" del intercambio de datos — cualquier sistema puede leerlo.
Anatomía de un CSV
producto,region,unidades,ingreso,fecha Laptop_Pro,Norte,145,217500,2024-03-01 Mouse_Inalambrico,Norte,820,16400,2024-03-01 Monitor_27,Sur,96,28800,2024-03-02
¿Para qué sirve?
Intercambio entre sistemas — exportar de un sistema e importar a otra herramienta sin perder compatibilidad.

Alimentar modelos — cargar datasets a Python, R, o Excel.

ETL — punto de salida de un sistema y entrada al siguiente.
¿Para qué NO sirve?
Datos anidados — para eso existe JSON.

Millones de filas — usa Parquet o BigQuery.

Tipos de datos — el CSV no distingue fechas de texto.

Tiempo real — el CSV es estático.

¿Por qué un feed de publicidad usa XML?
Cada producto tiene datos anidados (variantes, precios por región, imágenes) que un CSV plano no puede expresar. XML los estructura con etiquetas y es validable — Google Merchant Center rechaza el feed si falta un campo requerido.
<item> <id>SKU-4821</id> <title>Tenis Runner X Pro</title> <price currency="MXN">1299.00</price> <variant size="40" color="negro" stock="12"/> </item>

Infraestructura web

HTTP y APIs

HTTP es el protocolo que mueve datos por internet. Las APIs son los contratos que definen qué datos puedes pedir y cómo. Juntos, son la infraestructura invisible detrás de cada producto digital.

🌐 HTTP
🔌 API
HTTP (HyperText Transfer Protocol): el idioma que usan tu navegador (o celular) y un servidor para hablarse. Sin él, no podrías pedirle nada a internet ni internet podría responderte.
Imagínalo como pedir comida en un restaurante
Tú (el cliente) le haces una petición al mesero (el servidor): "quiero los tacos". El mesero va a la cocina, prepara lo que pediste, y te lo trae de vuelta. HTTP es exactamente esa conversación, pero entre tu teléfono y una computadora en algún centro de datos: cada vez que abres una app o una página, tu dispositivo le "pide algo" a un servidor y el servidor le "responde".

Los 4 métodos que vas a ver siempre
GET — pedir información, sin cambiar nada. "Muéstrame mis pedidos."

POST — crear algo nuevo. "Registra este nuevo pedido."

PUT / PATCH — actualizar algo que ya existe. "Cambia la dirección de envío."

DELETE — borrar. "Cancela este pedido."
La misma idea, en la vida real
GET = revisar el menú del restaurante.

POST = hacer un pedido nuevo.

PUT = pedirle al mesero que cambie algo del pedido.

DELETE = cancelar el pedido antes de que salga.
Anatomía de una petición y su respuesta
// Lo que el cliente envía (petición) GET /pedidos/45 HTTP/1.1 Host: tienda.com Authorization: Bearer token123 // Lo que el servidor regresa (respuesta) HTTP/1.1 200 OK Content-Type: application/json { "pedido": 45, "estado": "enviado" }
GET /pedidos/45 = método + qué quieres. Host = a quién le hablas. Authorization = tu credencial. La respuesta trae el código de estado y el dato pedido en JSON.
¿Y el "S" de HTTPS? Es HTTP con una capa de cifrado (la "S" es de "Secure"). Sin ella, cualquiera en la misma red podría leer lo que viaja entre tu dispositivo y el servidor — contraseñas, tarjetas, todo. Por eso hoy casi toda la web usa HTTPS por defecto; verlo en la barra del navegador (🔒) significa que esa conexión va cifrada.
Códigos de estado — la respuesta corta del servidor

Cada vez que pides algo, el servidor responde con un número de 3 dígitos que te dice, de un vistazo, si todo salió bien o algo falló — sin que tengas que leer un párrafo entero.

API (Application Programming Interface): el menú de un restaurante, pero para sistemas. No te deja entrar a la cocina (a la base de datos directamente) — te dice exactamente qué puedes pedir y en qué formato te lo van a servir.
¿Por qué existen las APIs?
Sin una API, para que dos sistemas se comuniquen tendrías que darle a uno acceso directo a la base de datos del otro — algo lento, inseguro y frágil. La API es la capa intermedia: define con claridad qué información puedes pedir, cómo la pides, y qué te va a devolver, sin que necesites saber cómo está armado el sistema por dentro.
Respuesta típica de una API — formato JSON
Casi todas las APIs modernas responden en JSON: texto organizado en pares de "etiqueta: valor", fácil de leer tanto para una persona como para un programa.
// Respuesta de una API de campañas { "campaign": "Lanzamiento_Producto_MX", "metrics": { "clicks": 45230, "impressions": 1200000, "ctr": 0.0377, "conversions": 892 } }
¿Para qué sirve en la práctica?
Automatizar reportes — un script llama a la API y el dashboard se actualiza solo, sin exportar manualmente.

Conectar sistemas — tu CRM le pregunta a la API de WhatsApp "¿llegó un mensaje?" y lo registra automáticamente.

Datos en tiempo real — métricas al minuto, no al día siguiente.

Esto es lo que un script envía y lo que la API devuelve. La URL es la "dirección" del recurso, el header lleva tu credencial, y el cuerpo de la respuesta llega en JSON listo para procesar.

¿Cuándo usar cada uno?
Usa Looker Studio cuando: tu stack es Google o necesitas compartir dashboards gratis.

Usa Power BI cuando: tu stack es Microsoft o necesitas análisis avanzados con DAX.

En la práctica: muchos equipos usan ambos — Power BI para análisis interno, Looker Studio para reportes compartidos.
